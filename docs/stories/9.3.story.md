## Story

**As a** desenvolvedor do EA360,
**I want** o Sentry configurado e capturando erros não tratados do frontend e do backend Next.js,
**so that** erros em produção sejam detectados, rastreados e corrigidos antes de impactar os usuários.

## Acceptance Criteria

1. Pacote `@sentry/nextjs` instalado e inicializado conforme documentação oficial para Next.js App Router.
2. `sentry.client.config.ts` e `sentry.server.config.ts` presentes e configurados com DSN via variável de ambiente `SENTRY_DSN`.
3. Erros não tratados no cliente (browser) são capturados automaticamente pelo Sentry.
4. Erros não tratados no servidor (Server Components, Server Actions, Route Handlers) são capturados automaticamente.
5. `SENTRY_DSN` está documentada no `.env.example` como variável opcional (Sentry é marcado como `opcional` no PRD §7).
6. `npm run build` conclui sem erros após a adição do Sentry.
7. `npm run lint` e `npm run typecheck` passam sem erros.

## Dev Notes

- Deliverable D-049; Épico E-XACC; PRD §5 "Observabilidade: Sentry + analytics de produto"; PRD §7 `@sentry/nextjs` (opcional).
- Prioridade PRD: Could — implementar sem bloquear outros épicos.
- Sentry é marcado como opcional no PRD; a aplicação DEVE funcionar normalmente se `SENTRY_DSN` não estiver definida.
- [AUTO-DECISION]: usar `@sentry/nextjs` com wizard para garantir compatibilidade com App Router, pois a configuração manual é propensa a erros para Next.js 15.
- Source maps: configurar upload de source maps no Sentry apenas se `SENTRY_AUTH_TOKEN` estiver disponível (também opcional).
- Não adicionar Sentry a rotas públicas sem auth (GOM browser) para evitar captura de dados de usuários não autenticados desnecessariamente.
